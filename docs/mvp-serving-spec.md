## 1. 먼저 정할 것 — 스코프 결정 (BLOCKING)

`architecture.md`는 **앱인토스를 알림 주력 채널**로 잡았다(푸시 가능하니까). 그런데 이 문서의 토폴로지는 **카톡 챗봇 + 알림톡**을 1차 채널로 가정한다. 둘은 공존 가능하지만 **1차 출시 대상**을 못박아야 이후 설계가 안 흔들린다.

- [ ] **결정 필요**: M1/1차 출시 채널 = (A) 카톡 챗봇+알림톡 / (B) 앱인토스 / (C) 둘 다
  - 코어(`server/kokalim/core`)는 채널 무관이라 어느 쪽이든 재사용됨 → 결정 대상은 "먼저 붙일 발송/조회 채널"일 뿐.
  - 이 문서 나머지는 **(A) 카톡 중심**을 가정하고 작성. (B)를 택하면 `apps-in-toss.md`가 정본.

## 2. 합의된 런타임 사실 (전제 — 재논의 금지)

이전 논의에서 확정된 제약. 설계는 이걸 깔고 간다.

1. **codex 백엔드는 연구실 PC에 고정**된다. 로컬 ChatGPT 로그인 + `codex exec` subprocess(타임아웃 180초)가 필요해서 Vercel/서버리스에선 못 돈다. codex를 안 쓰고 Anthropic API로 가면 유료(종량제)로 바뀐다.
2. **발송(알림톡/텔레그램 API 호출)은 그냥 outbound HTTP**다 → PC든 Vercel이든 어디서 트리거해도 된다. "누가 트리거하냐"만 정하면 됨.
3. **Vercel Hobby(무료) Cron = 하루 1회 + 시간대 부정확**(예: `0 8 * * *`이면 08:00~08:59 사이 아무 때). 더 자주/정밀하면 Pro. (출처: Vercel Cron 공식 문서)
4. **카카오 채널**: 챗봇(스킬)은 유저 발화에 **응답만** 가능(능동 push 불가). 단 **알림톡으로는 능동 발송 가능** — 친구 아니어도 발송, **전화번호 기반**, **정보성 템플릿 사전 심사** 필수(광고성 불가), 보통 카카오 파트너/딜러사 경유 + **건당 과금**. 챗봇 + 알림톡 **병행 가능**.
5. **"LLM 크론"은 유저별이 아니라 공유 1개**다. 유저 등록 = `watchlist`에 DB 한 줄 쓰기(즉시, LLM 없음). 크론이 주기적으로 `watchlist`를 읽어 **종목/사건당 1회** LLM 판정 후 구독자에게 fan-out. (유저 수에 LLM 비용이 비례하면 설계 실패 — `architecture.md §2.4`)
6. **Supabase = 공유 DB.** PC 워커와 Vercel이 둘 다 여기에 붙는다. 데이터의 단일 원천(single source of truth).

### 토폴로지 (합의)

```
                      ┌──────────────┐
  카톡 유저 ──발화──▶ │   Vercel     │  공개 HTTPS (항상 켜짐)
  (토스 유저)         │  - 카톡 webhook 응답(조회)
                      │  - 로그인/콜백
                      │  - 웹뷰 REST (앱인토스 택할 시)
                      │  - 브리핑 Cron (하루1회, 무료)
                      └──────┬───────┘
                             │ read/write
                      ┌──────▼───────┐
                      │  Supabase    │  DB (단일 원천)
                      └──────▲───────┘
                             │ read/write (outbound only)
                      ┌──────┴───────┐
  뉴스 소스 ──────▶   │  연구실 PC   │  비공개 (inbound 없음)
  (네이버/RSS)        │  - 수집 + codex 판정 (LLM 크론)
                      │  - 실시간 속보 발송(알림톡/텔레그램)
                      │  - 정밀 스케줄러(선택)
                      └──────────────┘
  발송: 알림톡/텔레그램 API는 outbound HTTP → PC 또는 Vercel Cron이 호출
```

핵심: **연구실 PC는 inbound 공개가 필요 없다**(밖으로 나가기만) → Cloudflare Tunnel/포트포워딩 불필요, 공격면↓.

## 3. 설계 영역별 백로그

### A. 로컬 PC — LLM 워커 (핵심)

**구현할 것**
- [ ] 속보 워커(poller): `watchlist`의 감시 종목 합집합 → 수집(`core/ingest`) → 중복제거(`core/dedup`) → **codex 판정(`core/triage`, 종목/사건당 1회)** → `events` 저장 → fan-out → 발송 → `deliveries` 기록. (현행 `pipeline.py`/`pipeline_context.py`를 DB 경로로 전환)
- [ ] 상시 기동: `systemd`(크래시 자동재시작) + UPS(정전 방어).
- [ ] codex 사용량 감시(`core/usage.py` 활용) → 한도 근접 시 텔레그램 경보.

**결정 필요**
- [ ] 수집 주기(현행 5분 유지?) / 종목당 발송 스로틀(같은 기업 연속 소식 병합?)
- [ ] LLM 백엔드 정책: codex(무료·한도·ToS 리스크) 기본 + 실패 시 Anthropic 폴백? (→ §H 시크릿 연동)
- [ ] "등록 즉시 오늘 정보 시드"를 지원할지 (지원 시 §F 조율 필요)
- [ ] PC 다운 감지/알림 방법(§J)

### B. Vercel — 공개 API + 크론

**구현할 것**
- [ ] 카톡 스킬 webhook 엔드포인트: 발화 → `intent` 파싱 → DB 조회 → 응답(5초 내). **LLM 동기 호출 금지**(DB 조회만).
- [ ] 알림 등록/해제 API: `watchlist` UPSERT (§2.5).
- [ ] 브리핑 Cron(하루 1회): DB 읽어 대상 유저에게 발송 트리거.
- [ ] (앱인토스 택할 시) 토스 로그인 콜백 + 웹뷰 REST.

**결정 필요**
- [ ] **런타임/언어**: 현재 백엔드는 Python(FastAPI). Vercel은 Node 1급 → (a) Python 서버리스로 통일 / (b) Node/TS로 짜고 도메인 로직 중복 감수 / (c) Vercel엔 얇은 프록시만 두고 실제 로직은 PC. → 코드 중복 최소화 방안 확정.
- [ ] 브리핑 정밀도: 무료 Cron(하루1회·부정확)로 충분한가? 유저별 분 단위 시각이면 §I로.
- [ ] 콜드스타트 vs 카톡 5초 타임아웃 대응(warm-up/경량화).

### C. Supabase — DB (테이블은 확정, 보강 필요)

> 유저가 "테이블은 끝났다"고 함. 단 이 논의에서 새로 생긴 요구 때문에 **보강이 필요**하다. 아래는 추가 검토 대상.

**결정 필요 (스키마 보강)**
- [ ] **알림톡용 필드**: `identities` 또는 유저 프로필에 `phone`(전화번호) + `alimtalk_consent_at`(수집·발송 동의 시각). 카톡 user id만으론 알림톡 발송 불가(§2.4, §D).
- [ ] **유저별 발송 시각**: `watchlist` 또는 user_settings에 `briefing_time` 등.
- [ ] **PC↔Vercel 조율용 job/큐 테이블**(§F 채택 시): 예 `jobs(id, type, payload, status, created_at)`.
- [ ] `situations` 테이블 DDL 확정(`architecture.md §5`에 언급만 있고 정의 없음 — `core/context/situation.py` payload 기준).
- [ ] (선택) 카톡 대화 로그 테이블(CS·어뷰징·분석 필요 시).
- [ ] 커넥션: Supabase pooler(6543) 사용 + 커넥션 재사용(무료 티어 커넥션 한도).

### D. 카카오톡 연결

**구현할 것**
- [ ] 카카오톡 채널 개설 + 챗봇(스킬) 서버 연결(webhook = §B).
- [ ] 알림톡: 비즈메시지 API 연동(파트너/딜러사) + 정보성 템플릿 등록·심사.
- [ ] 전화번호 수집·동의 플로우: 챗봇 유입 → 전화번호 제공/동의 → 알림톡 발송 대상 등록(§C 필드).

**결정 필요**
- [ ] 알림톡 발송 경로: 카카오 공식 파트너/딜러사 선정(직접 비즈메시지 API vs 대행 SDK).
- [ ] **뉴스 알림 템플릿 설계**: 헤드라인을 변수로 넣는 정보성 템플릿(토스 스마트발송 `apps-in-toss.md §5.3`와 동일 구조). 예: `{기업명} 새 소식: {헤드라인}` + 상세 링크. 심사 통과 문구 확정.
- [ ] 건당 과금 단가/예산(§K).
- [ ] 챗봇 인텐트 목록: "오늘 소식", "내 콕리스트", "알림 켜기/끄기" 등 → 응답 포맷.

### E. 인증 · 아이덴티티 (놓치기 쉬움)

**구현할 것**
- [ ] `identities`로 채널별 유저를 단일 `user_id`에 매핑: 카톡 user id / (앱인토스면 토스 userKey) / 전화번호.
- [ ] 세션: 웹뷰 쓸 경우 자체 JWT(`apps-in-toss.md §4`).

**결정 필요**
- [ ] 전화번호(알림톡)와 카톡 user id를 **어떻게 같은 유저로 연결**하는가(챗봇에서 전화번호 인증/수집 절차).
- [ ] 개인정보 최소 수집 원칙: 어디까지 저장(전화번호는 PII → 보존·파기 정책).

### F. PC ↔ Vercel 조율 (놓치기 쉬움)

Vercel(공개)과 PC(비공개, inbound 없음)가 협업하는 방법.

**결정 필요**
- [ ] 기본 방식: **Supabase를 메시지 버스로**(Vercel이 `jobs`에 write → PC가 폴링해서 처리). PC가 inbound를 안 여는 전제와 맞음.
- [ ] "등록 즉시 시드" 트리거를 이 방식으로 할지 / 아예 지원 안 할지.
- [ ] 발송 주체 분담: 실시간 속보 = PC(5분·codex), 브리핑 = Vercel Cron(하루1회). 겹치지 않게 소유권 명확히.

### G. 발송 채널 어댑터 (놓치기 쉬움)

**구현할 것**
- [ ] `channels/` 어댑터(공통 `Notifier` 프로토콜, `restructuring.md Step3`): `alimtalk`, `telegram`, (`toss_push`), `console`.
- [ ] `deliveries` 기반 재발송 방지 + 실패 로그.

**결정 필요**
- [ ] 케이스별 채널 매핑: 실시간 속보 → 알림톡? 텔레그램? / 브리핑 → 알림톡 / 운영 경보 → 텔레그램.
- [ ] 알림톡 실패(미동의·전화번호 없음) 시 폴백 정책.

### H. 시크릿 · 환경변수 (놓치기 쉬움)

**결정 필요 — 각 시크릿이 어디에 사는가**
- [ ] PC: codex 로그인, `ANTHROPIC_API_KEY`, 텔레그램 토큰, 네이버 API, Supabase 서비스키.
- [ ] Vercel: Supabase 키(가능하면 anon/제한 롤), 카톡 스킬 검증 키, (앱인토스면 토스 mTLS — 서버리스 mTLS 제약 주의), JWT_SECRET.
- [ ] 알림톡 API 키(파트너) 위치.
- [ ] 공용 연구실 PC면 시크릿 파일 권한 잠금 + 전용 계정. `.env` 커밋 금지 유지.

### I. 스케줄 · 타이밍 정책 (놓치기 쉬움)

**결정 필요**
- [ ] 속보 주기(5분?) 확정.
- [ ] 유저별 브리핑 시각 정밀도: 무료 Vercel Cron(하루1회·부정확)로 충분 / 분 단위 정밀 필요 시 → **PC 스케줄러(1분마다 DB 체크)** 또는 Vercel Pro.
- [ ] per-user 발송 스로틀(알림톡 과다 발송 방지 + 카카오/토스 rate limit).

### J. 모니터링 · 운영 · 장애대응 (놓치기 쉬움)

**구현할 것**
- [ ] 텔레그램 운영 채널: 파이프라인 오류·발송 통계·codex 한도 경보.
- [ ] PC 다운 감지(heartbeat) → 경보.

**결정 필요**
- [ ] PC 다운 시 서비스 영향 범위(조회는 Vercel로 생존, 신규 수집·속보 정지) 수용 기준.

### K. 비용 (놓치기 쉬움)

**결정 필요 — 예산 파악**

- [ ] Supabase/Vercel: 무료 티어 한도 vs 유료 전환 시점.
- [ ] 도메인.

### L. 배포 · 기동 (놓치기 쉬움)

**구현할 것**
- [ ] PC: `pip install -e server` → `systemd` 유닛(워커) + UPS.
- [ ] Vercel: 프로젝트 연결 + Cron 설정 + 환경변수.
- [ ] 도메인/HTTPS(카톡 webhook·(웹뷰) 오리진).

### M. 테스트 · 검증

**구현할 것**

- [ ] DB 기반 fan-out/재발송 방지 테스트(`deliveries`).
- [ ] 카톡 webhook·알림톡 발송 모킹 테스트.

## 4. 착수 순서 제안 (크리티컬 패스)

1. **§1 스코프 결정** (카톡 중심 확정) — 이게 안 되면 나머지 흔들림.
2. **§D 알림톡 파트너/템플릿 심사 착수** — 검수 리드타임 있음, 개발과 병렬 최우선.
3. **§C 스키마 보강**(전화번호·동의·발송시각·job큐) → **§B 등록/조회 API + §A 워커 DB 전환**.
4. **§F 조율 + §G 발송 어댑터** → **§B 브리핑 Cron**.
5. **§H/§I/§J/§L 운영 스펙** 채우고 → **§M 검증** → 소규모 출시.

## 부록: 이 문서가 담은 이전 논의 결론

- DB는 채널 공용 코어이며 봇 전용이 아니다(앱인토스도, 카톡도 유저 상태를 DB로 관리).
- 로컬 SQLite=개발용, 운영은 네트워크 Postgres(=Supabase 등). Supabase는 "관리형 DB 한 종류"일 뿐 필수 아님.
- 백엔드(FastAPI + 상시 poller)는 서버리스와 안 맞음 → 상시 프로세스(PC/클라우드) 필요. Vercel은 공개 창구·크론용.
- codex는 PC 고정 → "Vercel이냐 PC냐"는 위치가 아니라 "무료 codex vs 유료 API"의 문제이기도 하다.
- 카톡 능동 알림은 **알림톡으로 가능**(전화번호+정보성 템플릿 심사). 챗봇은 조회형, 알림톡은 능동 발송, 병행 가능.
- LLM 크론은 공유 1개. 유저 등록=DB write, 판정=종목/사건당 1회 후 fan-out.
